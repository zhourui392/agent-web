开启「RAG 召回」开关后,每发一条消息都会拿它去历史结论向量库里捞相似的过往经验, 命中就把「历史参考」拼到发给 CLI 的消息前面。本页观测的就是这个「捞」的过程跑得好不好—— 哪些消息捞到了、捞得准不准、链路有没有报错。

顶部 KPI
召回尝试
开关开着时每条消息触发一次;空消息 / 斜杠命令会算「跳过」,不去检索。
有效执行
真正跑完向量检索的次数(命中 + 0命中 + 异常),不含跳过 / 处理中。下面几个率多以它为分母。
质量命中率
命中 ÷ (命中 + 0命中)。真去检索时多大比例能捞到可用历史参考——最该盯的指标。低 = 库里没相关内容或阈值太严。
0 命中率
检索跑通但没有一条 chunk 过阈值,消息按原样发。
异常率
embedding 接口 / 向量计算报错的比例。已降级,不影响消息正常发送。
服务可用率
(命中 + 0命中) ÷ 有效执行 = 1 − 异常率,检索链路本身没崩的比例。
平均命中数
命中的那些尝试里,平均拼进去几条历史参考。
平均耗时
有效执行的平均墙钟耗时(含 embedding 往返)。
状态(每条尝试的最终态,即表格「状态」列)
命中
至少 1 条历史 chunk 过了所有阈值,已拼进 prompt。
0 命中
检索跑通但没 chunk 过阈值,prompt 不变。
异常
检索过程报错、已降级;消息照常发出,只是没注入历史参考。
跳过
没真去检索——空消息、斜杠命令,或召回开关关着。
处理中
终态还没写回,一般一闪而过。
分数与计数(表格「Top / 过滤」列、Score 样本、详情)
vector / topVector
query 与 chunk 的余弦相似度(0~1),越高语义越近。须先过「余弦硬闸」(约 0.6) 才进入打分。
final / topFinal
融合分 = 0.7·余弦 + 0.2·关键词重合 + 0.1·时间衰减·原始分。排序和最终去留看它。
below
被余弦硬闸刷掉的 chunk 数(语义不够近)。
ranked
过了余弦硬闸、参与融合排序的候选数。
active / filtered
active=库里活跃 chunk 总数;filtered=按来源 / tier 过滤后进入打分的候选数。
topK
最多注入几条(取融合分最高的前 K)。
badVector
向量损坏 / 计算报错被跳过的 chunk 数,正常应为 0。
三个面板 & 筛选
质量分桶
按 环境 / 模型 / 来源(CHAT·DIAGNOSE) / tier 分组各自的命中率,横向比较哪个维度召回质量差。
Score 样本
最近若干条尝试的 topVector / topFinal 实际分,用来校准阈值设得合不合理。
Top chunks
被召回次数最多的历史结论,看哪些经验最常被复用。
sourceType
历史来源:CHAT 普通会话沉淀 / DIAGNOSE 诊断结论。
tier
可信级别:VERIFIED 已验证 / PENDING 待定 / EXPLORATORY 探索性。
